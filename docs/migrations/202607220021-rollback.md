# Rollback: attendance groups and legacy work schedules

The contract migration must only be rolled back during a maintenance window. The authoritative pre-change rows are stored in `public.migration_snapshots` under migration key `202607220019`; the snapshot table is readable only by `super_admin` and `it_admin` and writable only by `service_role`.

Rollback order:

1. Stop writes from the web application and pause the DeviceGateway worker.
2. Recreate `attendance_groups`, `work_schedules`, and `schedule_rules` from migration `202606220001`.
3. Re-add `employees.attendance_group_id`, `daily_attendance.schedule_id`, and `departments.branch_id` with their original foreign keys.
4. Restore rows with `jsonb_populate_recordset` from the corresponding snapshot rows:

   ```sql
   insert into public.attendance_groups
   select restored.* from public.migration_snapshots snapshot
   cross join lateral jsonb_populate_recordset(null::public.attendance_groups, snapshot.rows) restored
   where snapshot.migration_key='202607220019' and snapshot.table_name='attendance_groups';
   ```

   Repeat for `work_schedules` and `schedule_rules`, then restore employee/group and daily/schedule references from `employee_attendance_groups` and `daily_attendance_schedules`.

5. Restore the legacy `attendance_report_rows`, `get_attendance_daily_report`, and `admin_save_employee` definitions from migrations `202607210012` and `202607220015`–`018`.
6. Deploy the previous Edge Functions/web commit, resume the worker, and verify counts against each snapshot's `row_count`.

Do not delete `attendance_report_rules` or `daily_attendance.rule_id` during rollback; they are additive and preserve the new rule provenance.
